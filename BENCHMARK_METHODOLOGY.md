# Benchmark Generation and Methodology

## Purpose

The benchmark measures how well the Medium-article RAG assistant satisfies the assignment requirements using only the indexed Medium dataset context. It separates three concerns:

1. Retrieval quality: whether the right articles appear in retrieved context.
2. Answer correctness: whether the assistant answers the task.
3. Faithfulness: whether answer claims are supported by retrieved context.

## Benchmark Generation

Benchmark cases are generated by:

```sh
uv run python scripts/generate_ai_benchmark.py --config configs/experiments/dense_mmr.yaml --case-count 25
```

The generator reads the dataset path from the selected experiment config. For the 300-sample experiments, this is:

```text
data/medium-300-sample.csv
```

Generation uses OpenRouter, not the main RAG answer model:

```text
OPENROUTER_API_KEY
OPENROUTER_MODEL       optional, default: nvidia/nemotron-3-ultra-550b-a55b:free
OPENROUTER_API_BASE    optional, default: https://openrouter.ai/api/v1
```

The selected experiment config still controls which dataset path is sampled, but not the benchmark generation model.

The generated benchmark is written to:

```text
eval/medium_300_ai_benchmark.jsonl
```

## Case Mix

The AI-curated benchmark targets 25 cases:

```text
8  precise_fact_retrieval
5  multi_result_topic_listing
6  key_idea_summary
4  recommendation_with_evidence
2  unknown_or_unanswerable
```

Each case includes:

```text
case_id
article_id
title
url
authors
tags
source_article_ids
question
expected_answer
evidence_quote
question_type
generator
```

For `multi_result_topic_listing`, cases may also include:

```text
expected_titles
```

These expected titles are retrieval diagnostics only. They are not the only valid answer set.

## Running the Benchmark

Run a benchmark against a config:

```sh
uv run python scripts/run_ai_benchmark.py --config configs/experiments/dense_mmr.yaml --output eval/dense_mmr_results.jsonl
```

This writes:

```text
eval/dense_mmr_results.jsonl
eval/dense_mmr_results.summary.json
```

The benchmark judge also uses OpenRouter, not the main RAG provider:

```text
OPENROUTER_API_KEY
OPENROUTER_MODEL       optional, default: nvidia/nemotron-3-ultra-550b-a55b:free
OPENROUTER_API_BASE    optional, default: https://openrouter.ai/api/v1
```

The RAG pipeline itself still uses the configured embedding/chat provider and Pinecone:

```text
LLMOD_API_KEY
PINECONE_API_KEY
```

## Retrieval Metrics

The benchmark reports classical retrieval metrics on the final context order sent to the model:

```text
hit@1
hit@3
hit@5
hit@k
mrr@k
recall@k
```

For single-article tasks, relevance is based on:

```text
source_article_ids, or article_id/title fallback
```

For listing tasks, `expected_titles` are used only to compute retrieval diagnostics:

```text
expected_title_recall_at_k
expected_titles_hit_at_k
```

Unknown/unanswerable cases have no expected relevant document, so retrieval metrics are `null`.

## Answer Correctness

For standard single-answer categories, the judge checks whether the answer captures the expected answer without requiring exact wording:

```text
precise_fact_retrieval
key_idea_summary
recommendation_with_evidence
unknown_or_unanswerable
```

For `multi_result_topic_listing`, exact matching against `expected_titles` is intentionally not required. A listing answer passes when:

```text
- exactly 3 titles are returned
- titles are distinct
- each title appears in retrieved context
- each title is relevant to the requested topic
```

This allows different valid triples of retrieved articles to pass.

## Faithfulness

Faithfulness is judged against retrieved context only:

```text
faithful = every factual claim in the answer is supported by retrieved context
```

The result keeps both fields:

```text
faithful
grounded
```

`grounded` is a backward-compatible alias for `faithful`.

## Summary Metrics

Each `.summary.json` includes:

```text
answer_accuracy
faithfulness_rate
grounded_rate
mean_judge_score
retrieval_hit_rate
retrieval_metrics
question_type_metrics
```

For listing cases, additional metrics are reported:

```text
listing_valid_rate
listing_count_ok_rate
listing_distinct_rate
listing_context_title_rate
listing_relevance_rate
```

## Methodology Notes

- The same benchmark JSONL must be reused across configs.
- Reindex only when dataset, chunking, embedding model, namespace, or embedded text changes.
- Pure retrieval-strategy comparisons such as dense vs dense+MMR can reuse the same index if namespace and chunking are unchanged.
- Metrics evaluate the final retrieved context order, including any MMR reordering.
- Listing correctness validates against retrieved context only, not the full dataset.
- `expected_titles` for listing cases diagnose retrieval recall, not answer correctness.
